# Spec/implementation sanity assessment

This report compares `docs/spec.md`, `docs/content/architecture.md`, and the current Ruby implementation as of `0abc77b`. It is intentionally documentation-only: it identifies drift and suggests ways to unify the documents and implementation, but it does not implement fixes or create follow-up issues.

## Evidence

- Required docs read: `AGENTS.md`, `README.md`, `docs/spec.md`, `docs/content/architecture.md`, and `docs/deterministic-testing.md`. The requested `docs/architecture.md` path does not exist; `README.md` links architecture at `docs/content/architecture.md`.
- Implementation surfaces inspected: `lib/durababble.rb`, `lib/durababble/workflow.rb`, `lib/durababble/durable_object.rb`, `lib/durababble/engine.rb`, `lib/durababble/workflow_execution.rb`, `lib/durababble/workflow_step_runner.rb`, `lib/durababble/workflow_replay_history.rb`, `lib/durababble/workflow_determinism.rb`, `lib/durababble/store*.rb`, `lib/durababble/rpc_*.rb`, `lib/durababble/workflow_rpc.rb`, `lib/durababble/worker*.rb`, and `sig/durababble.rbs`.
- Reproduction commands: `mise exec -- bundle exec ruby -Ilib -e 'require "durababble"; ...'` confirmed `Workflow.start`/`Workflow.enqueue` accept only `input`, `store:`, and `engine:`, `DurableObject.at` accepts but does not persistently use `worker_pool:`/`idempotency_key:`, and no `patched`/`deprecate_patch` runtime API is present.
- Storage grep: `rg -n "CREATE TABLE IF NOT EXISTS .*\\b(leases|idempotency_keys|object_sleeps)\\b|worker_pool|lease_token" lib/durababble/store lib/durababble` found no persisted `worker_pool`, `lease_token`, `leases`, `idempotency_keys`, or `object_sleeps` schema.
- Feature grep: `rg -n "patched\\(|deprecate_patch|PayloadTooLarge|CircuitBreakerOpen|object_sleeps|cancel_sleep|relocate_worker_pool|def (list|find|pause|resume|destroy!|evict)\\b" lib test docs/spec.md docs/content` found these surfaces only in spec/docs or unrelated method names such as `Engine#resume`.

## Spec-only or ahead of implementation

| Spec commitment | Current implementation status | Suggested unification |
| --- | --- | --- |
| `Workflow.enqueue/start(input, engine:, id:, idempotency_key:, worker_pool:)` and idempotent workflow starts scoped by worker pool/class/operation/argument fingerprint. | `Workflow.enqueue` and `Workflow.start` only accept `input`, `store:`, and `engine:`; `Engine#enqueue` always generates a new workflow id and there is no workflow-start idempotency table. | Either implement `id`, `idempotency_key`, and `worker_pool` on workflow creation with a real `idempotency_keys` table, or revise the spec to describe the current simpler enqueue API. |
| Persisted worker-pool boundary on every durable target. | `WorkerRuntime` has a `worker_pool` label, but persisted `workflows`, `inbox`, `target_activations`, `durable_objects`, and sequences are keyed without `worker_pool`; workers avoid wrong classes through registry filters rather than durable pool isolation. | Decide whether worker pools are a durable routing boundary now. If yes, add `worker_pool` to creation APIs, persisted target keys, claim filters, activations, inbox, and object state; if no, downgrade the spec to registry-scoped workers. |
| Unified `leases` table keyed by `(worker_pool, target_kind, target_class, target_id)` with owner gRPC address, `lease_token`, and `lease_until`. | Workflow leases are columns on `workflows`; object ownership is inferred from running inbox rows; activation/outbox locks have their own `locked_by`/`locked_until`; no `lease_token` exists. | Either create the unified leases abstraction and migrate workflow/object/routing paths onto it, or amend the spec to state that the prototype uses per-table leases without lease tokens. |
| Sticky placement, object/workflow cache entries, `LeaseRenewer`, idle eviction, and lease-token-based cache invalidation. | `WorkerRuntime` sets `worker_id` to the RPC address and can receive `DeliverMessage`, but there is no owner cache, lease renewer, lease token, idle release loop, LRU capacity, or object instance cache. | Move sticky placement/cache language into a future-production section unless cache ownership is the next implementation target. |
| Workflow commands execute against the live workflow owner at a safe deterministic yield point. | `WorkflowRef` enqueues a workflow inbox row, but `Engine#drain_workflow_inbox` creates a fresh workflow instance and invokes the command method outside the running `#execute` instance and deterministic scheduler. | This is a direct semantics decision: either implement command delivery into the active deterministic execution, or rewrite the spec to define workflow commands as external control/query handlers separate from orchestration state. |
| Workflow `expose` methods are owner-local transient RPCs and reject `idempotency_key:`. | `WorkflowRef` runs exposed query methods on a new local instance and passes kwargs through; it does not route through `CallTransient` or validate an active owner. | Route exposed workflow queries through the same lease-aware transient RPC path as gRPC, or document them as local handle helpers with no owner-local semantics. |
| Durable object transient `expose` methods are owner-local shared reads that stop entering once mailbox work is waiting. | `DurableObjectRef` loads persisted state and invokes exposed query methods locally; no shared owner read gate or active-owner RPC is involved, although state mutation is explicitly blocked for exposed queries. | Either implement object owner-local `CallTransient` dispatch and read gating, or simplify the spec to persisted-state queries. |
| Durable object lifecycle `on_create`, `on_load`, `on_destroy`, pending `sleep_until`, `cancel_sleep`, `object_sleeps`, and sleep-to-inbox conversion. | `on_wake(payload:)` is invoked for existing wake inbox rows, but there is no public object sleep API, no `object_sleeps` table, and no create/load/destroy lifecycle dispatch. | Keep `on_wake` as implemented and move the rest to future scope, or add object sleep/lifecycle APIs plus conformance tests before documenting them as current contract. |

## Direct divergences to prioritize

1. **Worker-pool durability is the largest structural gap.** The spec treats worker pool as part of durable target identity and the routing/failover boundary, but the implementation only uses runtime labels and class registries. This affects workflow start shape, object identity, inbox keys, target activations, leases, idempotency scope, and gRPC routing.
2. **Workflow command semantics differ materially.** The spec says commands are delivered into the live deterministic workflow at safe yield points, while the implementation invokes exposed commands on a fresh workflow instance while draining inbox rows. Any command expected to inspect or mutate workflow-local orchestration state will not behave like the spec.
3. **Transient exposed methods are local persisted-state helpers, not owner-local RPC.** Workflow and object exposed queries do not route to the active owner through `CallTransient`, so the spec’s owner-local read/concurrency semantics are currently ahead of the code.
4. **Patch markers are documented as current durable replay machinery but do not exist.** Because patch markers are central to safe workflow code evolution, this should not remain in the current guarantee matrix unless implemented.
5. **The storage model mixes old and new designs.** The implementation has `workflow_history`, `inbox`, `mailbox_sequences`, and `target_activations`, but lacks `leases`, `idempotency_keys`, `object_sleeps`, and persisted `worker_pool`; it also retains `durable_object_commands`.
6. **Production-operation features are described too strongly for the prototype.** Payload limits, circuit breakers, Spiffe/mTLS policy, object management operations, CLI repair surfaces, sticky caches, and retention/partitioning are not implemented.

## Suggested unification plan

1. Split the spec into “implemented prototype contract” and “planned production contract.” Keep the guarantee matrix limited to implemented behavior that has tests, and move patch markers, unified leases, durable worker pools, object sleeps, object management, payload limits, circuit breakers, sticky caches, and full CLI operations into a future-contract section.
2. Decide the worker-pool model before making more runtime changes. If pool isolation is required, make it part of workflow/object creation, storage primary keys or indexes, inbox idempotency scope, target activations, and RPC routing in one coherent migration. If it is not required yet, document the current registry-filtered worker model and remove durable pool claims from the active guarantee matrix.
3. Reconcile exposed method semantics. Either implement owner-local `CallTransient` for workflow/object `expose` and scheduler-integrated workflow command delivery, or rewrite the spec to describe exposed queries as persisted-state reads and workflow commands as external inbox handlers.
4. Remove or document transitional schema. `durable_object_commands` should either be dropped as dead compatibility state or intentionally documented as legacy. Conversely, if `leases`, `idempotency_keys`, and `object_sleeps` remain in the spec, migrations should create and use them.
5. Align public API docs and RBS with actual signatures. The spec should mention supported `store:` kwargs if they are retained, and `Workflow.start/enqueue` should either implement or stop advertising `id:`, `idempotency_key:`, and `worker_pool:`.
6. Make query-registry requirements match enforcement. The current query-plan suite enforces hot-path coverage, not every SQL statement. The spec should say that, or the implementation should finish moving production SQL through `StoreQueries`.
7. Keep the current strengths explicit. The implementation already has meaningful durable workflow replay, retries, heartbeats, waits, cancellation, Paquito storage, inbox/activation ordering, object command atomicity, outbox/fences, backend conformance tests, deterministic simulation, and real gRPC transport tests. The spec should preserve those as the current correctness core while clearly labeling broader platform features as not yet part of the prototype.
